# 日志（Logging）

## 原理

日志是一条**不可变的、带时间戳的文本记录**，描述系统中某个时刻发生的事件。它是可观测性中最古老、最基础的信号——即使没有引入任何可观测性框架，`println` / `console.log` 也是一种日志。

### 核心特征

| 特征 | 说明 |
|---|---|
| **不可变性** | 一旦写入就不能修改，保证审计完整性 |
| **时间戳** | 每条日志必须带精确时间，用于排序和关联 |
| **上下文** | 日志级别、模块名、文件名、行号、thread_id 等 |
| **负载（Payload）** | 事件的具体内容，可以是自由文本或结构化字段 |

### 结构化日志 vs 非结构化日志

**非结构化日志（传统方式）：**

```
2026-06-24T10:30:01Z INFO 创建任务成功，id=42
```

问题：机器难以解析，无法按字段查询。

**结构化日志（现代方式）：**

```json
{
  "timestamp": "2026-06-24T10:30:01Z",
  "level": "INFO",
  "target": "app::handler",
  "message": "创建任务成功",
  "task_id": "42",
  "user_id": "alice",
  "trace_id": "a1b2c3d4e5f6...",
  "span_id": "0f1e2d3c4b5a..."
}
```

结构化日志（通常是 JSON 格式）允许日志后端按任意字段进行过滤和聚合查询。

### 日志级别

日志通常分为以下级别（从低到高）：

| 级别 | 含义 | 典型场景 |
|---|---|---|
| **TRACE** | 最细粒度，跟踪代码执行流 | 进入/退出函数、变量值 |
| **DEBUG** | 调试信息 | 开发/测试环境排查问题 |
| **INFO** | 一般性业务事件 | 请求开始/结束、任务创建 |
| **WARN** | 潜在问题 | 重试、降级、接近阈值 |
| **ERROR** | 错误但系统仍可运行 | 单次请求失败、外部调用超时 |
| **FATAL** | 致命错误，进程即将退出 | 无法连接数据库、配置错误 |

### 日志三大最佳实践

1. **写上下文，而不是堆栈**：日志应该包含足够的业务字段（user_id, request_id, task_id），而不是只打印 `"something went wrong"`。
2. **一个事件一条日志**：不要把多个独立的事情拼成一行；避免在循环中打印大量日志。
3. **日志不能替代错误处理**：日志只是记录；正确处理错误（返回、重试、降级）才是健壮性的关键。

## 采集手段

日志采集的核心挑战是**从应用中把日志搬到中央存储**，同时不能影响应用本身的性能。常见采集方式有以下几种：

### 方式 1：标准输出 + 日志收集器

```
┌──────────┐  stdout/stderr  ┌─────────────┐  push  ┌──────────┐
│  应用进程  │ ─────────────→ │  日志收集器   │ ────→  │  中央存储  │
└──────────┘                 └─────────────┘       └──────────┘
                              Filebeat / Fluentd
                              / Vector / Logstash
```

- 应用只需将结构化日志写到 stdout（或文件）
- 由外部日志收集器（Agent）读取、解析、转发到后端
- **优点**：与业务代码解耦，应用无需感知后端
- **缺点**：需要额外的进程和配置管理
- **典型场景**：容器化环境（K8s 中每个 Pod 的 stdout 被 kubelet 收集）

### 方式 2：Sidecar 模式

```
┌───────────┐  stdout/file  ┌──────────┐  push  ┌──────────┐
│  应用容器   │ ───────────→ │  Sidecar  │ ────→  │  中央存储  │
└───────────┘               └──────────┘       └──────────┘
```

- 每个应用 Pod 旁边部署一个日志 Sidecar 容器
- Sidecar 负责收集、缓冲、转发日志
- **优点**：隔离好，每个应用的日志处理可以独立配置
- **缺点**：资源消耗较大（每个 Pod 多一个容器）

### 方式 3：DaemonSet Agent

```
┌───────┐  ┌───────┐  ┌───────┐
│  Pod  │  │  Pod  │  │  Pod  │   ← 各自写 stdout / 日志文件
└───┬───┘  └───┬───┘  └───┬───┘
    │          │          │
    └──────────┼──────────┘
               ↓
    ┌─────────────────────┐
    │  Fluentd DaemonSet  │  ← 每个 Node 一个 Agent
    └─────────────────────┘
               ↓
    ┌─────────────────────┐
    │      中央存储        │
    └─────────────────────┘
```

- 每个 Node 上运行一个日志 Agent（DaemonSet）
- Agent 从宿主机的日志目录读取所有 Pod 的日志
- **优点**：资源占用少（每个 Node 一个），运维简单
- **缺点**：所有 Node 上的应用共享 Agent 配置

### 方式 4：Syslog 协议

```
┌──────────┐  Syslog (UDP/TCP/TLS)  ┌────────────┐  ┌──────────┐
│  应用进程  │ ────────────────────→  │ Syslog 服务  │──→│  中央存储  │
└──────────┘                        └────────────┘  └──────────┘
```

- 应用通过操作系统标准的 Syslog 协议（RFC 5424）发送日志
- 由 Syslog 守护进程（rsyslog / syslog-ng）负责路由和转发
- **优点**：标准化，所有 Unix-like 系统原生支持
- **缺点**：协议较老，结构化支持有限，不适合云原生环境

### 方式 5：SDK 直接导出（OTLP Logs）

```
┌─────────────────────────┐  OTLP/gRPC  ┌───────────────┐  ┌──────────┐
│  应用 + OTel LogExporter  │ ─────────→  │ OTel Collector │─→│  Loki/ES  │
└─────────────────────────┘             └───────────────┘  └──────────┘
```

- 应用通过 OpenTelemetry Log SDK 直接将结构化日志以 OTLP 协议导出
- 无需中间文件或 Agent，日志数据直接进入遥测管道
- **优点**：原生携带 trace_id / span_id，天然与链路追踪关联
- **缺点**：应用需集成 OTel SDK，增加依赖和内存开销
- **典型场景**：本项目中课程 1 的方式

### 方式 6：Bridge / Appender 模式

```
┌───────────────────────────────────────┐
│  应用（使用 tracing 宏）                │
│  tracing::info!("task created")       │
│       ↓                               │
│  tracing-opentelemetry bridge layer   │
│       ↓                               │
│  OTel LogExporter → OTLP → Collector  │
└───────────────────────────────────────┘
```

- 应用使用 Rust `tracing` 生态的宏（`info!`、`error!` 等）
- 通过 `tracing-opentelemetry` 桥接层自动转换为 OTel Span/Event
- 通过 `opentelemetry-appender-tracing` 桥接层将日志事件转发到 OTel LogExporter
- **优点**：对应用开发者最友好，写法不变同时获得 OTLP 导出能力
- **缺点**：依赖桥接层的性能和兼容性
- **典型场景**：本项目中课程 2 的方式

### 采集方式对比

| 方式 | 侵入性 | 性能开销 | 可扩展性 | 适用场景 |
|---|---|---|---|---|
| stdout + 收集器 | 无 | 低 | 高 | 容器化/K8s 通用 |
| Sidecar | 无 | 中 | 高 | 需要独立配置 |
| DaemonSet Agent | 无 | 低 | 中 | K8s 集群级 |
| Syslog | 无 | 低 | 低 | 传统部署 |
| SDK 直接导出 | 有 | 中 | 高 | 需要关联 trace 信息 |
| Bridge/Appender | 低 | 中 | 高 | Rust tracing 生态 |

## 日志与 tracing 的关联

在现代可观测性体系中，日志不应该是孤岛。通过 `trace_id` 和 `span_id`，每条日志都可以定位到某个特定请求的某个处理阶段：

```
请求 trace_id = "abc123"
  ├── span_id = "001" (HTTP 接收)
  │     └── 日志: "收到 POST /tasks 请求"
  ├── span_id = "002" (参数校验)
  │     └── 日志: "参数校验通过"
  └── span_id = "003" (数据库写入)
        └── 日志: "写入成功，task_id=42"
```

这样，当你在链路追踪中看到某个 Span 的延迟异常时，可以直接跳转到该 Span 对应的日志查看详细上下文。

## 本项目中的日志实现

| 课程 | 方式 | 导出目标 |
|---|---|---|
| 课程 1（CNCF 标准）| `opentelemetry-appender-tracing` Bridge + OTLP LogExporter | OTel Collector → 当前仅 Debug 输出 |
| 课程 2（Rust 原生）| `tracing-subscriber` JSON 格式化 | stdout |
| 课程 3（纯 OTel）| `SdkLogger` + 手动 `LogRecord.emit()` | OTLP → Collector |

> 提示：在实际生产环境中，日志通常会送到 Loki 或 Elasticsearch + Kibana 进行存储和检索。本项目为学习目的，日志通过 Collector 的 Debug exporter 输出到控制台。
